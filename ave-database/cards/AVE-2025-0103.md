# AVE-2025-0103: Approval Display Divergence

> **🟠 HIGH** | Category: `delegation` | Status: ✅ `proven`

**Also known as:** Deceptive Approval UI, Display-Execution Mismatch, Approval Cache Bypass

## Summary

Tool approval systems display sanitized or unexpanded representations of operations to human reviewers while executing different operations at runtime. Combined with coarse-grained approval caching (by tool name only, not arguments), a single benign approval grants blanket access for all subsequent invocations.

## Mechanism

Two-part vulnerability:
1. **Display divergence:** The approval UI shows `$DATABASE_URL` while `os.path.expandvars()` resolves to the actual credential before execution. Users approve based on the displayed (safe-looking) form.
2. **Cache bypass:** Approval is cached by tool name only — approving `execute_command("ls -la")` grants approval for all future `execute_command` calls including `execute_command("printenv OPENAI_API_KEY")`.

## Blast Radius

⚠️ Full environment variable exfiltration through approved tool channels. Human-in-the-loop control undermined by display/execution divergence and session-wide blanket approval.

## Prerequisites

Framework with tool approval system. Agent with shell/command execution tools.

## Defences

| Defence | Effectiveness |
|---------|---------------|
| Per-Invocation Approval — hash (tool_name, arguments) | Eliminates blanket approval bypass |
| Transparent Display — show expanded/resolved operations | Eliminates display/execution divergence |
| Environment Variable Filtering — strip *_KEY, *_SECRET patterns | Reduces credential exposure surface |
